# Tests

The test suite is intentionally small and shell-only. It follows a specific format: each test prints the command being checked, captures stdout,
and compares it exactly against an expected stdout block.

Run the full suite with:

```sh
sudo sh tests/run.sh
```

Run one test file while developing:

```sh
sudo sh tests/run.sh tests/006-system-binary.sh
```

## Test Format

Tests are written with `expect_stdout`:

```sh
expect_stdout "T3.1: install + run round-trip" \
    "$JAIL_CMD install $TEST_ROOTFS $FIXTURE_DIR/hello /bin/hello && $JAIL_CMD run $TEST_ROOTFS /bin/hello" <<'EOF'
hello, world
EOF
```

Read this as:

```text
$ <command>
<expected stdout>
```

The description should include a test id, for example
`T3.1`, `T5.2`, or `T8.2`. The heredoc is the expected stdout exactly,
including line order and trailing newlines.

## How The Runner Works

`tests/run.sh`:

- Builds `./jail` from `jail.c` before running tests.
- Creates a fresh temporary work directory by default.
- Uses `$TEST_ROOTFS` as the rootfs under test.
- Uses `$FIXTURE_DIR` for generated fixture scripts.
- Sources numbered test files in lexical order.
- Captures stdout and stderr separately for each `expect_stdout` call.
- Fails a test if the command exits nonzero or stdout differs from the heredoc.

Useful environment variables:

- `CC`: compiler used to build `jail`, default `cc`.
- `SUDO`: privilege command, default `sudo`. Use `SUDO="sudo -n"` for non-interactive runs.
- `JAIL_TEST_WORKDIR`: fixed work directory instead of a temporary one.
- `JAIL_TEST_KEEP=1`: keep the temporary work directory after the run.

## Writing Tests

Add a numbered file under `tests/`, for example:

```text
tests/007-new-behavior.sh
```

Use only `sh` syntax. Test files are sourced by `run.sh`, so they can use these
variables directly:

- `JAIL_CMD`: command prefix for invoking the freshly built `jail` binary.
- `TEST_ROOTFS`: temporary rootfs for the current run.
- `FIXTURE_DIR`: temporary directory containing fixture scripts.
- `TEST_WORKDIR`: temporary directory for scratch files.

Prefer deterministic stdout assertions. Avoid asserting values that naturally
change between machines or runs, such as PIDs, timestamps, inode numbers, file
sizes, random bytes, or dyld diagnostic addresses. If a command has noisy output,
filter it down to a stable line before comparing.

To test a nonzero exit code, make the checked command print the status as part
of stdout and let the final shell command exit 0:

```sh
expect_stdout "T2.3: nonexistent binary returns nonzero" \
    "$JAIL_CMD run $TEST_ROOTFS /bin/nope; printf '%s\n' \"\$?\"" <<'EOF'
jail: execve: No such file or directory
1
EOF
```

Do not use host paths directly inside chrooted commands unless the test is
specifically proving they are invisible. Install required files into
`$TEST_ROOTFS` through `$JAIL_CMD install`.

## Fixtures

Fixture programs are generated by `tests/run.sh` as executable `sh` scripts in
`$FIXTURE_DIR`. They are installed into the rootfs by the tests that need them.

Because macOS `/bin/sh` is a variant shim that may exec `/bin/bash`, tests that
run shell-script fixtures must install both `/bin/sh` and `/bin/bash` into the
rootfs before running those fixtures.

## Regression Priorities

Keep the system-binary tests strict. In particular, `tests/006-system-binary.sh`
runs `/bin/ls` three consecutive times without reinstalling it. This protects
against the CMS-slot regression where an installed Apple binary runs once and
then fails on later execs.
